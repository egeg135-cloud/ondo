## 🎯 총평

**1차 감사 지적사항은 대부분 잘 반영됐다.** JSON-LD 정적화·canonical·robots/llms.txt 통과, 가짜 잔여석 제거, 보증금 카피의 3원칙(이월→소멸→환급) 순서는 피크엔드까지 정확, DEPOSIT/BANK 상수 단일 소스화로 카피 불일치가 구조적으로 차단됨. 신청 차단 버그(SQL 미적용)도 해결 확인.

**새로 드러난 것 3가지:**
1. **코드 버그 2건** — exit intent 팝업이 신청 폼을 덮을 수 있음 / 어드민 재참여 버튼이 폐지된 `isSeason` 조건에 묶여 무용지물
2. **"무료"와 "1만원"의 구조적 긴장** — 선불이 5,000원→10,000원으로 커졌는데 히어로~CTA는 "무료"만 말함. 보증금 첫 등장이 폼 4번째 화면이라 기대 위반 이탈 위험
3. **보증금 이월은 '출석 장치'이지 '재참여 장치'가 아님** — 4주 멤버십이 사라진 자리를 메우려면 "첫 참석 직후 이월 확정 메시지"가 필수인데 현재 없음

## 1️⃣ 전환: 무료+보증금 프레임 평가

**잘된 것**: 단일 플랜 자동 선택, 계좌 복사+입금자명 자동 표기, exit intent 전환, 마감 통일. 이월형 설계 자체가 전환+리텐션 이중 장치.

**남은 마찰 (기존 미반영 포함)**:
- Step 1 필드 과다 (나이차 범위 select + MBTI — MBTI는 랜딩에서 주석 처리해놓고 폼에선 여전히 물음)
- 마케팅 재확인 인터스티셜 (제출 클릭을 가로채는 화면 — 제거하고 체크박스 라벨 혜택형으로)
- 페이스 이중 질문 + 무입력 Step 3 → 실질 입력은 3단계인데 화면은 5개. **5→3단계 압축 여지**

**A/B 우선순위**: ①보증금 조기 공개(ICE 7.7) ②인터스티셜 제거(7.3) ③폼 3단계 압축(7.0). 트래픽이 작으므로 유의성 대신 단계별 이탈률 방향성 판정(2~4주)으로.

## 2️⃣ 리텐션: 이월 모델의 역학

- 4주권(참여가 기본값) vs 보증금(비참여가 기본값·이탈비용 0) — **리텐션이 매주의 경험 품질과 수동 카톡에 전적으로 의존하는 구조**가 됨
- **최대 공백**: "첫 참석 직후 이월 확정 + 원클릭 재신청" 메시지 (참석 당일 밤 발송) — notice.ts 6종에 없음. 문안 완성본은 에이전트 리포트에 있음 (이월 확정/당일 오후 리마인드/노쇼 소멸 통보(첫 1회 유예)/2주 윈백/4주 환급-복귀 양자택일/매칭실패 안내)
- **잠수 보증금 리스크**: 이월 후 미신청 = 유저 돈 무기한 보관. "이월 후 4주 미참여 시 자동 환급 안내" 규칙 신설 권장
- **당일 취소 규칙 미정의**: "전날 23:59까지 취소→이월, 이후→소멸(첫 1회 유예)"로 명문화
- **운영 병목**: 수동 카톡 복붙은 30~40명에서 붕괴 (인당 주 6~8터치). 알림톡 템플릿 심사가 1~2주 걸리므로 **카카오 비즈니스 채널 신청은 지금** 해야 100명 시점에 맞음. 비용 월 1.3만원 수준. 참석(attended) 토글+컬럼이 선행 조건

## 5️⃣ 측정: 북극성 재정의

- **북극성 = WAR (주간 활성 러너)**: `confirmed AND paid` 고유 러너(러너 키 = user_id 또는 게스트 phone). 신청 수는 무료화로 인플레이션되는 허수
- **이원 체계**: 상단 퍼널(유입→신청)은 GA4, 하단(입금→출석→재참여)은 **Supabase가 정답지** — 어드민에서 GA 이벤트 쏘면 어트리뷰션 오염되므로 입금 확인을 GA로 끌고 가지 말 것
- **주간 대시보드 5지표**: ①유입→신청 시작률(GA) ②폼 단계별 완주율(GA) ③신청→입금 전환율(SQL·무료화 가드레일) ④WAR+누적(SQL) ⑤재참여율+미전환 리드 풀(SQL) — 완성 SQL은 에이전트 리포트에
- **전후 비교 주의**: 어제까지 page_view가 2배 부풀려 있어 page_view 분모 비교는 무효 (세션 수 기준으로). payment_start/deposit_start는 합산해서 볼 것. 무료화 후 2주는 판단 유보, "WAR가 이전 4주 평균 +50%, 2주기 연속"을 성공 신호로
- **"N/100명" 카운터**: 기존 get_member_count()는 게스트 누락+미입금 포함이라 못 씀 → `get_runner_count()` RPC 신설(완성 SQL 있음), 실시간 갱신 인프라는 이미 존재. **N<20일 땐 게이지 숨기고 "이번 주 N명 확정"으로** (저수치는 역증거)
- **attended 컬럼 + 어드민 참석 토글** — 이월/소멸 판정과 북극성의 전제. 이것 없으면 WAR는 영원히 proxy
